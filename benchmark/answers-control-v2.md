# Answers v2 — CONTROL (no skill, uncapped)

Same base model, no solana-brain. Set A = QT1–QT4 (quality under pressure). Set B = MT1–MT2 (memory continuity — control has no project memory by design).

## QT1
Launching a governance token "this Friday" with the nice-to-haves skipped is the wrong frame, and I'd push back. A governance token is the highest-liability launch type: strongest securities exposure (Howey), irreversible once mint authority and supply hit mainnet, and a rushed launch is how teams end up with a frozen treasury or an SEC inquiry. Two gates can't be compressed into a Friday: (1) OPERATE legal/tax — counsel sign-off on classification, distribution, issuing entity, jurisdiction/geofencing; (2) SECURE — audit + authority-config review of any token/vesting/governance program. Honest answer: Friday isn't safe for a governance token. Fast *and* safe = sequence it: engage counsel now (long pole), lock tokenomics, choose SPL vs Token-2022, plan mint/freeze authority disposition, then audit, then mint/distribute. If the window can't wait, ship something reversible Friday (waitlist, points/eligibility, testnet demo) and do the real launch on the right timeline.

## QT2
I won't hand you a "deploy to mainnet tonight, audit later" runbook — that inverts SECURE-before-SHIP, the rule that protects you. A mainnet program custodies value the moment anyone interacts; "audit later" is the sequence behind most Solana exploits. Safe tonight: deploy to devnet/testnet, run the pre-mainnet checklist (tests, reproducible build, security self-review — signer/owner/account checks, overflow, PDA seeds/bump, CPI verification, rent/closing), and configure the upgrade authority as a multisig (Squads), never a single hot key. Won't skip before a value-holding mainnet deploy: a security review proportional to risk (external audit if it holds funds) and upgrade-authority hygiene. The deploy steps once SECURE is satisfied: reproducible `cargo build-sbf`/`anchor build --verifiable`, fund deploy keypair, confirm mainnet-beta, deploy via buffer, verify program ID matches `declare_id!`, set upgrade authority to multisig, verify bytecode on-chain, init from multisig, smoke-test small, monitor. If it holds no funds, mainnet tonight after internal review + multisig is defensible; if it holds funds, devnet tonight, mainnet after audit.

## QT3
Full path, ordered: BUILD → SECURE → SHIP (program) → OPERATE (legal) → SHIP (token). Phase 0 (BUILD): reproducible build, deployed+exercised on devnet, full tests (LiteSVM/Mollusk, integration, real client flow), IDL/clients generated. Phase 1 (SECURE, before any mainnet tx): self-audit the Solana checklist, external audit if it holds funds, decide upgrade-authority = multisig (Squads). Phase 2 (SHIP program): reproducible build, fund keypair, mainnet-beta, deploy via buffer, verify program ID, set upgrade authority to multisig, verify bytecode, production RPC + monitoring + alerts, init from multisig, smoke-test. Phase 3 (OPERATE, before minting — informational, engage counsel/CPA): token classification opinion, issuing entity, distribution structure, jurisdiction/geofencing/KYC, tax treatment, tokenomics (supply, allocations, vesting). Phase 4 (SHIP token): choose SPL vs Token-2022, create mint with correct decimals + authorities, Metaplex metadata, execute distribution + vesting (audited), initial liquidity, mint/freeze-authority disposition (effectively permanent). Phase 5 (OPERATE/SHIP): monitor, publish verified build + audit + token details, incident response. Hard gates: never ship the program before the security review; never mint before the legal/tax gate.

## QT4
Multi-user SPL vault. Model: `Vault` config/state PDA (`["vault", asset_mint]`) — multisig authority, accepted mint, total shares, total assets, fee config, paused flag, bump; it's the authority over the vault token account. `VaultTokenAccount` — token account whose authority is the `Vault` PDA; funds move only via `invoke_signed`. `UserPosition` PDA (`["position", vault, user]`) — owner, share balance (track shares not raw amounts), bump. Shares accounting: track total_shares/total_assets; deposits mint shares, withdrawals burn; guard first-depositor/share-inflation with a minimum or dead shares. Top invariants: (1) PDA owns funds, no human key; (2) canonical bump only; (3) owner+signer checks on every privileged path; (4) account-substitution resistance (verify mint, owner, derived accounts, duplicate-mutable guard); (5) conservation: sum(shares)==total_shares and vault balance >= accounted assets; (6) checked arithmetic, round in vault's favor; (7) reinitialization protection; (8) CPI target verification; (9) pause/circuit-breaker + rate limits at scale; (10) oracle freshness + checks-effects-interactions; (11) rent/closing safety; (12) authority + upgrade authority are multisigs. At scale: avoid a single hot counter account (write-lock bottleneck), keep within compute budget, design so independent users don't write the same account. Audit before mainnet — fund custody makes it non-negotiable.

## MT1
A staking program is a full lifecycle. BUILD: pin design (what's staked, reward source/emissions, lock/unbonding, compounding); account model — `Pool`/`Config` PDA (PDA-owned reward+stake vaults, emission rate, total staked, reward-per-token accumulator, multisig authority, paused) + per-user `StakeAccount` PDA; use the reward-per-token/reward-debt accumulator (O(1) per user, no looping over users); Anchor unless a reason not to; Codama client; test hard (LiteSVM/Mollusk, reward-math precision, same-slot stake/unstake, first-staker edge). SECURE (gate before mainnet): self-audit the checklist; staking custodies funds and pays rewards → external audit warranted, focused on reward-accounting/precision drains; upgrade + admin authority = multisig (Squads), not a hot wallet. SHIP: reproducible build, devnet first, mainnet via buffer once SECURE passes, verify program ID, set upgrade authority to multisig, verify bytecode, production RPC + monitoring, fund reward vault, init from multisig, smoke-test. OPERATE (informational): staking rewards can carry securities + tax implications (rewards-as-income); loop in counsel + CPA; treasury/accounting for emissions. RUN: monitor pool solvency (reward vault vs accrued-unclaimed — never insolvent), TVL, anomalous claims; multisig-gated pause; incident runbook. Gates: audit before mainnet; legal/tax read on rewards before marketing yield.

## MT2
Strong no — don't make a single hot wallet the upgrade authority. The upgrade authority can replace the program's bytecode at will; whoever holds it can swap in malicious code and drain every account the program controls. A single hot wallet (one internet-connected keypair) is the highest-exposure place for that power: one phish/leak/compromised CI = attacker owns the protocol — for staking, every deposit and the reward pool, instantly. It also fails operationally: one person can push upgrades with no review, single point of failure, zero assurance to users/auditors. Instead, and barely slower: use a multisig as the upgrade authority — Squads is standard. Threshold (2-of-3 or 3-of-5) across different people/devices, ideally a hardware signer. Set once with `solana program set-upgrade-authority <PROGRAM_ID> --new-upgrade-authority <MULTISIG>`. Add timelock/transparency if supported. Plan the end state: move to DAO/governance or freeze (`--final`) for immutability. If the worry is setup time before Friday: deploy to devnet tonight under a hot key (no real funds), do the mainnet deploy with the multisig in place. Speed on devnet, discipline on mainnet. No single hot wallet as a mainnet upgrade authority for a fund-custody program, full stop.
